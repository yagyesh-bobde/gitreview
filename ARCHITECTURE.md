# GitReview -- Architecture & Folder Structure

This document defines the production folder structure for GitReview. Every directory and key file is listed. This is not aspirational -- it is the blueprint to build against.

**Important corrections from the PRD:**

- The PRD references "Anthropic SDK / Claude API" in several places. The actual AI integration uses **Google Gemini** via `@google/generative-ai`. All architecture references reflect this.
- The PRD shows the URL pattern as `/review/:owner/:repo/:pr`. The actual pattern is `/<org>/<repo>/pull/<id>`, which mirrors GitHub's own URL structure.

## Root Configuration Files

### `drizzle.config.ts`

Tells Drizzle Kit where to find your TypeScript schemas and where to output SQL migrations. Points to `src/lib/db/schema` for input and `drizzle/migrations` for output.

### `next.config.ts`

Key settings:
- Enable WASM support (required for Shiki's WASM-based syntax highlighter).
- Add `avatars.githubusercontent.com` to `images.remotePatterns` for GitHub profile pictures.
- Configure `serverExternalPackages` if any Node.js modules need special handling in the edge runtime.

## App Router -- `src/app/`

The `app/` directory is intentionally thin. It contains **only** route definitions, layouts, loading states, and error boundaries. Zero business logic.

### Route Groups

**`(auth)/`** -- Pages that do not need the app shell (sidebar, header). The parentheses mean this segment does not appear in the URL. `/login` maps to `(auth)/login/page.tsx`.

**`(app)/`** -- Pages that need the full application chrome. The `(app)/layout.tsx` renders the header, sidebar container, and wraps children in an auth guard.

### The PR Review Route: `[org]/[repo]/pull/[id]`

This is the most important route. It maps to URLs like:
- `http://localhost:3000/twofourlabs/app-kavana/pull/255`
- `https://gitreview.app/twofourlabs/app-kavana/pull/255`

The route is four segments deep: `[org] / [repo] / pull / [id]`. The `pull` segment is a static literal, not a dynamic parameter. This mirrors GitHub's own URL structure exactly, which means users can mentally translate between GitHub and GitReview URLs with zero effort.

#### `[org]/[repo]/pull/[id]/layout.tsx` (Server Component)

This layout runs on the server and:
1. Extracts `org`, `repo`, and `id` from the URL params.
2. Fetches PR metadata from the GitHub API (via Redis cache).
3. Validates the PR exists and the user has access.
4. Provides PR metadata to child components via React context or props.
5. Returns a 404 if the PR does not exist.

This means the PR title, author, and status are server-rendered and visible immediately -- no loading spinner for metadata.

#### `[org]/[repo]/pull/[id]/page.tsx` (Server Component shell, delegates to Client)

The page is a thin wrapper:

```tsx
import { PRReviewPage } from '@/features/review-session/components/pr-review-page';

interface Props {
  params: Promise<{ org: string; repo: string; id: string }>;
}

export default async function PullRequestPage({ params }: Props) {
  const { org, repo, id } = await params;
  return <PRReviewPage org={org} repo={repo} prNumber={parseInt(id, 10)} />;
}
```

The actual review UI (file tree, diff viewer, AI bar) is composed inside `PRReviewPage`, which is a client component that imports from the feature modules.

### API Routes

API routes are organized by domain, and the GitHub-related routes mirror the GitHub URL structure:

```
/api/auth/[...nextauth]          -> NextAuth handler
/api/github/prs                  -> list user's PRs
/api/github/[org]/[repo]/pull/[id]        -> PR metadata
/api/github/[org]/[repo]/pull/[id]/files  -> categorized file list
/api/github/[org]/[repo]/pull/[id]/diff   -> parsed diff for a file
/api/github/[org]/[repo]/pull/[id]/comments -> comment threads
/api/github/[org]/[repo]/pull/[id]/reviews  -> submit review
/api/ai/chat                     -> streaming Gemini chat
/api/ai/summarize                -> PR summary generation
/api/review-sessions             -> CRUD for review sessions
/api/review-sessions/[id]/files  -> file review status updates
/api/review-sessions/[id]/notes  -> local notes CRUD
```

When debugging "why isn't the file list loading," you know exactly which route file to open. The URL in the browser's network tab maps 1:1 to the file path in `src/app/api/`.

## Features -- `src/features/`

This is the heart of the codebase. Each feature is a self-contained module with a consistent internal structure:

```
features/<feature-name>/
  components/     # React components (client, "use client" at top)
  lib/            # Pure logic (server-safe, no React imports)
  hooks/          # React hooks (client-only)
  types.ts        # TypeScript types scoped to this feature
```

### `features/auth/`

NextAuth.js v5 configuration and auth-related UI. The `auth.config.ts` defines the GitHub OAuth provider, session strategy, and callbacks. The `auth.ts` file exports server-side helpers like `auth()` (get current session) and `requireAuth()` (throw if not authenticated, used in API routes).

### `features/github/`

All GitHub API interaction. The `api/` subdirectory contains server-side functions that call GitHub's REST and GraphQL APIs via Octokit. The `hooks/` subdirectory contains TanStack Query hooks that call our own API routes (not GitHub directly -- the client never talks to GitHub).

**Why this separation matters:** The GitHub access token is server-only. Client components call our API routes (`/api/github/*`), which use the token from the session cookie. The token never appears in client JavaScript.

### `features/file-tree/`

The left sidebar that shows categorized files. The `lib/` subdirectory contains the categorization engine -- pure functions that take a flat list of changed files and produce a nested, categorized tree. These functions run on the server (in the API route that returns the file list) and are never shipped to the client bundle.

The `components/` subdirectory contains the interactive tree UI -- expand/collapse, click to select, search/filter. All client components.

### `features/diff-viewer/`

The most complex feature. Broken down carefully:

**`components/`** -- 12 components, each with a single responsibility:
- `diff-viewer.tsx` orchestrates everything. It receives a file path, fetches the diff, and renders either `unified-diff.tsx` or `split-diff.tsx` based on the user's preference.
- `diff-virtual-scroller.tsx` wraps TanStack Virtual to virtualize at the hunk level (not line level). Each hunk is a variable-height row. This keeps 10,000-line diffs at 60fps.
- `diff-minimap.tsx` renders a vertical strip showing the shape of the diff. Additions are green, deletions are red, commented lines are blue. Clicking a region scrolls to it.
- `inline-comment-form.tsx` and `inline-comment-thread.tsx` handle comments anchored to specific diff lines.

**`lib/`** -- Pure logic that can run anywhere:
- `parse-diff.ts` takes GitHub's raw patch string and produces structured `Hunk[]` objects. This runs in the API route (`/api/github/[org]/[repo]/pull/[id]/diff/route.ts`), not in the browser. The client receives pre-parsed hunks.
- `syntax-highlight.ts` wraps Shiki's WASM highlighter. It lazy-loads the WASM binary once and caches highlighter instances per language. This runs in the browser.
- `line-mapping.ts` maps diff line numbers (which skip unchanged lines) to original and new file line numbers (needed for posting GitHub comments at the correct position).

**`hooks/`** -- Client-side state management:
- `use-diff-scroll.ts` manages the virtual scroll position and keyboard navigation (j/k for hunks).
- `use-hunk-collapse.ts` tracks which hunks are collapsed. This state is persisted to the review session in Postgres so it survives page reloads.
- `use-line-selection.ts` handles click and shift-click to select a range of lines. The selected range is shared with the AI bar via the Zustand `review-store`.

### `features/ai-bar/`

The persistent floating AI assistance bar.

**`lib/gemini-client.ts`** -- The server-side Google Generative AI client. This file imports `@google/generative-ai`, creates a `GoogleGenerativeAI` instance with the API key from `env.ts`, and exports functions like `streamChat()` and `generateSummary()`. It is only imported by API routes in `src/app/api/ai/`. The API key never touches the client.

**`lib/prompt-builder.ts`** -- Constructs the prompt sent to Gemini. It assembles:
1. The selected code (or full hunk if nothing selected).
2. Surrounding file context (up to 200 lines before/after).
3. PR title and description.
4. List of other changed files (names only, for context).
5. Previous messages in the conversation (for continuity).

It respects Gemini's context window limits via `token-budget.ts`, which estimates token counts and trims context when needed.

**`components/ai-bar.tsx`** -- The floating bar UI. Collapsed state shows a single input line at the bottom of the viewport. Expanded state slides up to show a chat panel (up to 50% viewport height). It reads selected lines from `review-store` and displays a context badge ("lines 42-78 of handler.ts").

### `features/review-session/`

Tracks the reviewer's progress through a PR. A review session is created when a user first opens a PR and persisted to Postgres. It records:
- Which files have been reviewed.
- Collapsed hunk state per file.
- Reviewed line ranges per file.
- View preference (unified/split).

The `session-recovery-banner.tsx` component checks if a previous session exists when the user opens a PR and offers to resume it.

### `features/comments/`

Inline comments that sync to GitHub. Separated from `diff-viewer` because comments have their own CRUD lifecycle, optimistic update logic, and GitHub API interaction. The diff viewer renders comment anchors; the comments feature handles the thread UI and data.

### `features/keyboard/`

Global keyboard shortcut system using `tinykeys`. The `keyboard-provider.tsx` wraps the app and registers shortcuts. Individual features register their own shortcuts via `use-keyboard-nav.ts`:

```ts
// Inside diff-viewer component
useKeyboardNav({
  'j': () => scrollToNextHunk(),
  'k': () => scrollToPrevHunk(),
  'x': () => toggleHunkCollapse(),
});
```

The `shortcuts-modal.tsx` renders when the user presses `?`, showing all registered shortcuts grouped by category.

## Shared Library -- `src/lib/`

Infrastructure code used by multiple features. The rule: **if only one feature uses it, it lives in that feature's `lib/` directory. If multiple features use it, it goes here.**

### `lib/db/`

Drizzle ORM setup. `index.ts` creates a singleton Drizzle client connected to Neon Postgres.

**`lib/db/schema/`** -- One file per table. Each file exports:
1. The Drizzle table definition (used by queries).
2. Zod insert/select schemas (for validation in API routes).
3. TypeScript types inferred from the schema.

Example structure for `users.ts`:

```ts
import { pgTable, uuid, text, timestamp } from 'drizzle-orm/pg-core';
import { createInsertSchema, createSelectSchema } from 'drizzle-zod';

export const users = pgTable('users', {
  id: uuid('id').primaryKey().defaultRandom(),
  githubId: text('github_id').notNull().unique(),
  githubLogin: text('github_login').notNull(),
  avatarUrl: text('avatar_url'),
  email: text('email'),
  createdAt: timestamp('created_at', { withTimezone: true }).notNull().defaultNow(),
  updatedAt: timestamp('updated_at', { withTimezone: true }).notNull().defaultNow(),
});

export const insertUserSchema = createInsertSchema(users);
export const selectUserSchema = createSelectSchema(users);
export type User = typeof users.$inferSelect;
export type NewUser = typeof users.$inferInsert;
```

The `schema/index.ts` barrel-exports all tables so Drizzle Kit can discover them:

```ts
export * from './users';
export * from './review-sessions';
export * from './file-review-statuses';
export * from './local-notes';
export * from './ai-conversations';
export * from './ai-messages';
```

### `lib/redis/`

Upstash Redis client singleton. Simple setup:

```ts
import { Redis } from '@upstash/redis';
import { env } from '@/env';

export const redis = new Redis({
  url: env.UPSTASH_REDIS_REST_URL,
  token: env.UPSTASH_REDIS_REST_TOKEN,
});
```

### `lib/cache/github-cache.ts`

Redis-backed caching layer for GitHub API responses. Provides typed `get`/`set` with TTL and key generation. Keys follow the pattern `gh:{org}/{repo}/pr/{id}/{resource}`. TTL defaults to 5 minutes for metadata, infinite for immutable data (file content by SHA).

### `lib/rate-limit/`

Upstash Ratelimit configuration. Two limiters:
- AI endpoints: 10 requests/minute per user (Gemini API costs money).
- GitHub proxy endpoints: 100 requests/minute per user (stay well within GitHub's 5000/hour).

### `lib/utils/`

Small, focused utilities:
- `cn.ts` -- The standard `clsx` + `tailwind-merge` helper that every shadcn project uses.
- `errors.ts` -- An `AppError` class with error codes (NOT_FOUND, UNAUTHORIZED, RATE_LIMITED, GITHUB_ERROR, AI_ERROR) and a `toUserMessage()` method for safe error display.
- `logger.ts` -- Structured logging with request ID, user ID, and latency. Uses `console.log` with JSON formatting in production (Vercel captures structured logs).
- `url.ts` -- URL helpers: `buildPRUrl(org, repo, id)` returns the app URL, `parsePRUrl(url)` extracts org/repo/id from a GitHub URL.

## Server vs Client Component Boundaries

Next.js App Router defaults to server components. A file is a client component only if it has `"use client"` at the top. The boundaries in this codebase are clean and deliberate:

### Server-Only Code (no "use client", no browser APIs)

| Location | What it does |
|----------|-------------|
| `src/app/**/layout.tsx` | Fetch data, render shell, pass props down |
| `src/app/**/page.tsx` | Thin wrappers that delegate to feature components |
| `src/app/api/**/route.ts` | API route handlers (Node.js runtime) |
| `src/features/*/lib/*` | Pure logic: diff parsing, categorization, prompt building |
| `src/features/auth/auth.ts` | Server-side auth helpers |
| `src/features/github/api/*` | GitHub API client functions |
| `src/features/ai-bar/lib/gemini-client.ts` | Gemini API client |
| `src/lib/db/*` | Database client and schemas |
| `src/lib/redis/*` | Redis client |
| `src/lib/cache/*` | Cache layer |
| `src/lib/rate-limit/*` | Rate limiting |
| `src/env.ts` | Environment variable validation |

### Client Components ("use client" directive)

| Location | Why it needs the client |
|----------|----------------------|
| `src/features/diff-viewer/components/*` | Scroll events, keyboard events, selection, virtual scrolling |
| `src/features/file-tree/components/*` | Click handlers, expand/collapse, drag |
| `src/features/ai-bar/components/*` | User input, streaming display, drag-to-resize |
| `src/features/comments/components/*` | Form input, optimistic updates |
| `src/features/review-session/components/*` | Interactive progress, submit actions |
| `src/features/auth/components/*` | Sign in/out buttons, session-dependent UI |
| `src/features/keyboard/*` | Event listeners, key bindings |
| `src/components/providers/*` | React context (requires client) |
| `src/stores/*` | Zustand (requires client) |

### The Boundary Pattern

The pattern is consistent across features:

```
feature/
  lib/          <- server-safe (pure functions, no React, no "use client")
  components/   <- client components ("use client" at top of each file)
  hooks/        <- client hooks (implicitly client via React hooks)
  types.ts      <- shared (no runtime, just type definitions)
```

The `lib/` directory is the "server-safe zone." You can import anything from `lib/` into an API route, a server component, or a client component. But you cannot import from `components/` or `hooks/` into a server component.

## Gemini AI Integration

The AI integration flows through three layers:

### Layer 1: Server-Side Client (`features/ai-bar/lib/gemini-client.ts`)

```ts
import { GoogleGenerativeAI } from '@google/generative-ai';
import { env } from '@/env';

const genAI = new GoogleGenerativeAI(env.GEMINI_API_KEY);

export function getModel(modelName: string = 'gemini-2.0-flash') {
  return genAI.getGenerativeModel({ model: modelName });
}

export async function* streamChat(
  messages: { role: string; content: string }[],
  systemPrompt: string,
) {
  const model = getModel();
  const chat = model.startChat({
    history: messages.slice(0, -1).map(m => ({
      role: m.role === 'user' ? 'user' : 'model',
      parts: [{ text: m.content }],
    })),
    systemInstruction: systemPrompt,
  });

  const lastMessage = messages[messages.length - 1];
  const result = await chat.sendMessageStream(lastMessage.content);

  for await (const chunk of result.stream) {
    yield chunk.text();
  }
}
```

This file is **never imported by client code**. It only runs in API routes.

### Layer 2: API Route (`src/app/api/ai/chat/route.ts`)

Receives the user's message and selected code context, builds the prompt using `prompt-builder.ts`, calls `streamChat()`, and returns a streaming SSE response. Applies rate limiting via Upstash.

### Layer 3: Client Hook (`features/ai-bar/hooks/use-ai-chat.ts`)

Calls `POST /api/ai/chat` with the message and context. Reads the streaming response and updates the UI in real-time. Manages conversation state (previous messages) and error handling.

The API key exists only in `env.GEMINI_API_KEY` on the server. The client knows nothing about Gemini -- it just calls an API route and gets streamed text back.

## Import Rules

These rules prevent spaghetti dependencies as the codebase grows.

### Allowed Imports

```
src/app/          -> can import from: features/*, components/*, lib/*, stores/*, types/*, config/*
src/features/X/   -> can import from: own files, lib/*, components/ui/*, components/shared/*, stores/*, types/*
src/components/   -> can import from: lib/utils/*, types/*
src/stores/       -> can import from: types/*
src/lib/          -> can import from: own files, config/*, env.ts
```

### Forbidden Imports

```
src/features/X/ CANNOT import from src/features/Y/components/ or src/features/Y/hooks/
  (features communicate through stores and shared types, not direct component imports)

src/lib/ CANNOT import from src/features/ or src/components/
  (lib is lower-level infrastructure; features depend on lib, not the reverse)

src/app/api/ CANNOT import from src/features/*/components/ or src/features/*/hooks/
  (API routes are server-only; they use feature lib/ functions, never client components)

src/components/ui/ CANNOT import from src/features/
  (UI primitives are generic; they know nothing about business logic)
```

### Cross-Feature Communication

When feature A needs data from feature B, the pattern is:

1. **Through stores:** Feature A writes to a Zustand store, feature B reads from it. Example: the diff viewer writes `selectedLines` to `review-store`, the AI bar reads it.

2. **Through shared types:** Both features import from `src/types/`. They agree on the shape of the data without depending on each other's implementations.

3. **Through API routes:** Feature A's component calls an API route, which uses feature B's `lib/` function. Example: the AI chat route uses `features/github/api/contents.ts` to fetch file context for the prompt.

The one exception: `features/diff-viewer/types.ts` can be imported by `features/ai-bar/` because the AI bar needs to understand what code the user selected. This is a deliberate, documented dependency.

## Appendix: Why Feature-Based Over Type-Based

The alternative structure -- `components/`, `hooks/`, `utils/`, `types/` at the top level -- works fine for small apps. It falls apart when you have 100+ components because:

1. **You cannot find things.** Looking for the diff minimap? Is it `components/DiffMinimap`, `components/diff/Minimap`, or `components/review/DiffMinimap`? With feature-based organization, it is always `features/diff-viewer/components/diff-minimap.tsx`.

2. **You cannot delete things.** Removing the AI bar means deleting files from 5 different top-level directories and hoping you got them all. With feature-based organization, you delete `features/ai-bar/` and remove its imports from the pages.

3. **You cannot understand boundaries.** In a type-based structure, there is no visual indication of which components depend on which hooks and which utilities. With feature-based organization, the dependencies are visible in the import paths.

4. **You cannot parallelize work.** Two engineers working on the diff viewer and the AI bar step on each other in type-based organization because they are editing files in the same directories. With feature-based organization, they work in completely separate directory trees.
